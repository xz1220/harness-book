# 小红书选题 001：Harness 不是提示词工程

## 选题钩子

面试官问：“用过 Harness 吗？”

你回答：“不就是提示词工程吗？”

空气安静了。

## 核心 user story

你写了一个最简单的 AI Agent：

- 模型能理解任务
- 模型能调用 bash
- 模型能读文件、改文件
- 模型能根据观察继续下一步

第一天它看起来很聪明。

第二天你发现它会卡在一条永远不返回的命令里。

第三天你发现它会把十万行日志全塞回上下文。

第四天你发现它换了一个上下文窗口，就忘了刚才做到哪。

于是你开始加：

- timeout
- 工具路由
- 权限确认
- 输出截断
- 上下文压缩
- 进度文件
- 子 Agent

这些东西不是提示词。

它们也不是模型本身。

它们合起来，就是 Harness。

## 封面文案

主标题：

> Harness 不是提示词工程

副标题：

> 它是让 AI Agent 真正落地的系统工程

备选标题：

## 正文草稿

很多人第一次听到 Harness，会把它理解成“提示词工程的高级说法”。

但这两个东西不是一层。

Prompt 解决的是：你希望模型怎么想、怎么说、怎么格式化输出。

Harness 解决的是：当模型真的要做事时，外部系统如何接住它的动作。

比如模型说：“我要执行 `ls`。”

真正执行命令的不是模型，而是 Harness。

模型说：“我要调用某个工具。”

真正判断这个工具能不能调、怎么调、结果怎么回填上下文的，也是 Harness。

模型卡住了，谁来 timeout？

工具输出太长，谁来截断？

命令有风险，谁来弹权限确认？

上下文快满了，谁来压缩历史？

任务跑了半小时，谁来记录进度？

这些工程判断都不属于 prompt 本身。

它们是把“模型自己做不到什么”这件事，写成可执行代码。

所以一个最小的 Harness 至少包含三件事：

1. Loop：反复调用模型，让它一步一步做事。
2. Routing：把模型的 tool call 路由到真实工具。
3. Assumptions：把对模型能力边界的判断写进代码。

一句话：

> Prompt 让模型更会说，Harness 让 Agent 真能做。
